## 개요

User Journey는 자동화된 워크플로우입니다. 하나의 메시지를 보내는 대신, 여러 단계로 이루어진 경로를 만듭니다. 사용자는 Journey를 통해 흘러가며 대기, 메시지, 행동에 따른 분기를 경험합니다.

## 작동 방식

1. 사용자가 Journey에 진입함 (이벤트, 스케줄 또는 API)
2. 노드(단계)를 통해 이동함
3. 각 노드가 무언가를 함: 대기, 메시지 발송, 조건 확인
4. Journey를 완료하거나 종료 조건을 충족하면 종료됨

## Journey 구성 요소

### 필수 요소
- **시작 노드** - 사용자가 시작하는 곳
- **완료 노드** - 성공적인 종료
- **비정상 종료 노드** - 오류/타임아웃 종료
- **조건부 종료 노드** - 종료 조건으로 인해 종료

## 노드 유형

| 노드 | 목적 |
|------|------|
| Start | 진입점 |
| Delay | 계속하기 전 대기 |
| Message | 메시지 발송 |
| Event Branching | 사용자 이벤트에 따른 다른 경로 |
| Segment Branching | 사용자 속성에 따른 다른 경로 |
| Variant Branching | A/B 테스트 경로 |
| User Update | 사용자 속성 변경 |
| Event Trigger | 커스텀 이벤트 발생 |
| Complete | Journey가 성공적으로 완료됨 |
| Conditional Exit | 종료 이벤트로 인해 종료 |
| Abnormal Exit | 오류/타임아웃으로 인해 종료 |

## Journey 흐름

```
[Start] → [Delay] → [Message] → [Event Branching]
                                      ↓
                    이벤트 발생? ──예──→ [Complete]
                         ↓
                       아니오 → [Delay] → [Message] → [Complete]
```

## 특수 기능

### 인앱/인웹 메시지
Journey에는 다음과 같은 온사이트 메시지 노드가 포함될 수 있습니다:
- 사용자가 앱/사이트를 열 때까지 대기
- 특정 이벤트에 따라 트리거
- 트리거 시간 제한 있음

### 전환 추적
사용자가 목표를 완료하는지 추적:
- 전환 이벤트 정의
- 전환 윈도우 설정 (예: 7일)
- 분석에서 전환율 확인

### 테스트 모드
라이브 전에 Journey 테스트:
- 테스트 모드 활성화
- 화이트리스트에 사용자 추가
- 화이트리스트 사용자만 Journey에 진입

## 규칙

- 각 Journey에는 정확히 하나의 시작 노드가 있음
- 노드는 "next" 참조를 통해 연결됨
- 메시지 노드에는 성공 및 실패 경로가 있음
- 종료 조건은 주요 지점에서 확인됨
- 사용자는 한 번에 하나의 "위치"에만 있을 수 있음
