## 개요

발송 금지 시간은 Campaign에 "방해 금지" 윈도우를 설정합니다. 이 시간대에는 메시지가 발송되지 않습니다. 사용자의 수면 시간과 업무 시간을 존중합니다.

## 작동 방식

1. Campaign에 발송 금지 시간이 구성됨
2. 메시지가 발송되려 할 때 시스템이 현재 시간을 확인함
3. 발송 금지 윈도우 내에 있으면 설정에 따라 메시지가 처리됨
4. 메시지가 건너뛰어지거나 윈도우가 끝날 때까지 보류(지연)됨

## 발송 금지 시간 설정

| 필드 | 의미 |
|------|------|
| Start | 방해 금지 시간 시작 (시, 분) |
| End | 방해 금지 시간 종료 (시, 분) |
| Allow hold | 메시지를 지연할지 건너뛸지 여부 |

## 시간 형식

시간은 시와 분으로 설정합니다:
- 시: 0-23 (24시간 형식)
- 분: 0-59

예: 22:00 (오후 10시)에 시작, 08:00 (오전 8시)에 종료

## 동작 옵션

### 건너뛰기 (allow_hold = false)
- 메시지가 발송되지 않음
- 메시지가 재예약되지 않음
- 사용자가 이 특정 발송을 받지 못함

### 보류/지연 (allow_hold = true)
- 발송 금지 윈도우가 끝날 때까지 메시지가 보류됨
- 방해 금지 시간이 끝나면 메시지가 발송됨
- 사용자가 결국 메시지를 받음

## 예시

| 설정 | 동작 |
|------|------|
| 22:00 - 08:00, 건너뛰기 | 오후 10시와 오전 8시 사이 메시지가 발송되지 않음 |
| 22:00 - 08:00, 보류 | 메시지가 보류되어 오전 8시에 발송됨 |
| 09:00 - 18:00, 건너뛰기 | 업무 시간 동안 메시지가 발송되지 않음 |

## 시간대 고려사항

발송 금지 시간은 [로컬 시간대 발송](/ko/knowledge-base/concepts/local-timezone-delivery)과 동일한 시간대 우선순위를 사용합니다:

| 우선순위 | 출처 |
|----------|------|
| 1 (최고) | 사용자 시간대 |
| 2 | 디바이스 시간대 |
| 3 (기본값) | 기본 시간대 (Asia/Seoul) |

이는 사용자의 시간대가 알려진 경우 각 사용자의 로컬 시간에서 방해 금지 시간이 평가됨을 의미합니다.

## User Journey 메시지 노드

User Journey에서 메시지 노드는 약간 다른 형식을 가집니다:
- `deliveryForbiddenTiming.start.hours`와 `.minutes`
- `deliveryForbiddenTiming.end.hours`와 `.minutes`
- `deliveryForbiddenTiming.shouldDefer` (allow_hold와 동일)

## 메시지 건너뛰기 시점

메시지가 건너뛰어지는 경우 (`forbidden_timing_filter` 건너뛰기 유형):
- 현재 시간이 발송 금지 윈도우 내에 있음
- 그리고 allow_hold가 false (또는 shouldDefer가 false)

## 규칙

- 시간은 사용자의 시간대에서 평가됨
- 밤새 걸치는 윈도우가 올바르게 작동함 (예: 22:00 - 08:00)
- 각 Campaign/메시지가 다른 발송 금지 시간을 가질 수 있음
- User Journey 메시지 노드는 자체 설정이 있음

## 예외 사항

- 사용자 시간대를 알 수 없으면 프로젝트 시간대로 대체
- 보류된 메시지는 Journey 시간 초과 시 만료될 수 있음
- 이벤트 트리거 Campaign은 너무 오래 보류되면 컨텍스트를 잃을 수 있음
