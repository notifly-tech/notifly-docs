## 개요

이 가이드는 고객이 campaign, journey, segment를 설정할 때 가장 자주 하는 실수를 나열합니다. 런칭 전에 검토하여 문제를 피하세요.

---

## Campaign 설정 실수

### 1. Campaign 활성화 잊음

**실수:** Campaign을 만들었지만 "Draft" 상태로 둠.

**결과:** Campaign이 실행되지 않음. 메시지 발송 안됨.

**피하는 방법:**
- 메시지 발송 전 항상 상태 확인
- 검토 후 활성화 알림 설정
- 체크리스트 사용: 저장됨 → 검토됨 → 활성화됨

---

### 2. 과거 시간으로 예약

**실수:** 이미 지난 일정 시간 설정.

**결과:** Campaign 활성화 안되거나, (설정에 따라) 즉시 실행됨.

**피하는 방법:**
- 날짜와 시간 모두 재확인
- 타임존 차이 고려
- 활성화가 지연되면 일정 업데이트

---

### 3. 잘못된 타임존 설정

**실수:** 오전 9시에 예약했지만 잘못된 타임존.

**결과:** Campaign이 예상치 못한 시간에 실행 (오전 3시? 오후 6시?).

**피하는 방법:**
- Campaign 설정에서 타임존 확인
- 본인의 타임존이 아닌 사용자의 타임존 고려
- 먼저 소규모 그룹으로 테스트

---

### 4. 이벤트 이름 오타 (대소문자 구분!)

**실수:** 트리거 이벤트가 `Purchase`인데 SDK는 `purchase` 발송.

**결과:** 이벤트 불일치. Campaign 트리거 안됨.

**피하는 방법:**
- SDK/이벤트 로그에서 정확한 이벤트 이름 복사
- 이벤트 이름은 대소문자 구분
- 런칭 전 트리거 테스트

---

### 5. A/B 테스트 그룹이 100%가 아님

**실수:** 그룹을 40%, 40%, 15%로 설정.

**결과:** 유효성 검사 오류로 Campaign 저장 안됨.

**피하는 방법:**
- 비율 합계가 정확히 100%인지 항상 확인
- UI의 유효성 검사 피드백 활용

---

## Segment 실수

### 6. OR을 의미할 때 AND 사용

**실수:** `country = US AND country = Korea` (둘 다 해당하는 것은 불가능).

**결과:** Segment에 0명 일치.

**피하는 방법:**
- AND = 모든 조건이 참이어야 함
- OR = 어떤 조건이든 참이면 됨
- "US 또는 Korea 사용자"의 경우, 그룹 간 OR 사용

---

### 7. 속성 이름은 대소문자 구분

**실수:** Segment가 `Email`로 필터링하지만 속성은 `email`.

**결과:** 필드 이름이 맞지 않아 아무도 일치하지 않음.

**피하는 방법:**
- 사용자 데이터에서 정확한 속성 이름 확인
- 데이터 탐색기에서 속성 이름 복사
- 명명 규칙 일관성 유지

---

### 8. 과도하게 제한적인 조건

**실수:** 많은 AND 조건 결합.

**결과:** 거의 모든 사람이 필터링됨. 매우 작은 대상자.

**피하는 방법:**
- 단순하게 시작, 조건을 하나씩 추가
- 각 조건 후 segment 수 확인
- 모든 조건이 정말 필요한지 고려

---

### 9. 문자열 vs 숫자 불일치

**실수:** 나이가 문자열 "30"으로 저장되어 있는데 `age > 30` 필터링.

**결과:** 비교가 예상대로 작동하지 않음.

**피하는 방법:**
- 데이터 타입 파악
- 데이터 타입에 맞는 연산자 사용
- 알려진 사용자로 테스트

---

## 발송 규칙 실수

### 10. 너무 공격적인 빈도 제한

**실수:** 모든 campaign에서 하루 1개 메시지로 전역 제한 설정.

**결과:** 사용자가 중요한 메시지를 놓침 - 이전 campaign이 한도 도달.

**피하는 방법:**
- 한도 설정 시 모든 campaign 고려
- 특정 campaign에는 campaign 수준 fatigue 사용
- 전역 제한은 절대 최대값으로 예약

---

### 11. 발송 금지 시간이 모든 메시지 차단

**실수:** 발송 금지 시간을 오후 6시 - 오전 9시로 설정 (15시간 침묵).

**결과:** 사용자가 잘못된 타임존에 있으면 대부분의 메시지가 발송되지 않음.

**피하는 방법:**
- 발송 금지 시간을 합리적으로 유지
- 다양한 타임존의 사용자 고려
- 발송에 미치는 영향 테스트

---

### 12. 카카오에 SMS 대체 발송 잊음

**실수:** 카카오가 없을 수 있는 사용자에게 SMS 대체 발송 없이 알림톡 발송.

**결과:** 카카오 없는 사용자는 아무것도 받지 못함.

**피하는 방법:**
- 중요한 알림톡 메시지에는 항상 SMS 대체 발송 활성화
- 대체 발송용 SMS 템플릿 설정
- 카카오와 SMS 경로 모두 테스트

---

## Journey 실수

### 13. 테스트 모드 켜둔 채로 방치

**실수:** 테스트 모드가 활성화된 상태로 Journey 게시.

**결과:** 화이트리스트 사용자만 진입. 나머지는 모두 제외됨.

**피하는 방법:**
- 런칭 체크리스트에 "테스트 모드 비활성화" 추가
- 런칭 전 journey 설정 확인
- 런칭 후 진입률 모니터링

---

### 14. Journey의 무한 루프

**실수:** 사용자를 이전 단계로 반복적으로 보내는 journey 로직.

**결과:** 사용자가 멈추고, 잠재적으로 많은 메시지 수신.

**피하는 방법:**
- Journey 흐름에서 루프 검토
- 탈출 조건 추가
- 루프가 의도적이면 최대 반복 횟수 설정

---

### 15. 이벤트 분기에 타임아웃 없음

**실수:** 이벤트 분기가 영원히 이벤트를 기다림.

**결과:** 이벤트를 수행하지 않는 사용자가 journey에 무기한 머무름.

**피하는 방법:**
- 이벤트 분기에 항상 타임아웃 설정
- "무응답" 경로 추가
- 사용자가 행동하지 않으면 어떻게 되는지 고려

---

### 16. 탈출 조건 누락

**실수:** 사용자가 목표를 일찍 달성해도 journey를 떠날 방법 없음.

**결과:** 이미 전환한 사용자가 여전히 후속 메시지 수신.

**피하는 방법:**
- 탈출 이벤트 정의 (구매, 가입 등)
- Journey 전체에서 탈출 조건 확인
- 탈출 경로 테스트

---

## 채널 설정 실수

### 17. 만료된 푸시 인증서

**실수:** APNs 인증서 만료 전에 갱신하지 않음.

**결과:** 모든 iOS 푸시 알림 실패. 고장날 때까지 경고 없음.

**피하는 방법:**
- 인증서 만료일 추적
- 만료 1개월 전 캘린더 알림 설정
- 새 인증서 조기 갱신 및 업로드

---

### 18. 개발 vs 프로덕션 자격증명 혼동

**실수:** 샌드박스 APNs로 테스트 후, 같은 자격증명으로 라이브.

**결과:** 프로덕션에서 푸시 알림 실패 - "invalid token".

**피하는 방법:**
- 개발과 프로덕션 자격증명 분리 유지
- 설정 중인 환경 확인
- 런칭 전 프로덕션에서 테스트

---

### 19. 미인증 이메일 발신자

**실수:** 인증되지 않은 이메일 주소로 발송 시도.

**결과:** 이메일 실패 또는 스팸으로 분류.

**피하는 방법:**
- 프로젝트 설정에서 발신자 이메일 주소 인증
- 도메인 인증 완료 (SPF, DKIM, DMARC)
- 일관된 발신자 주소 사용

---

### 20. 국가 코드 없는 전화번호

**실수:** 국가 코드 없이 `123-456-7890`으로 전화번호 저장.

**결과:** 번호가 불완전하여 SMS 발송 실패.

**피하는 방법:**
- 항상 국가 코드 포함: `+1234567890`
- 수집 시 전화번호 형식 검증
- E.164 형식 사용

---

## 데이터 및 연동 실수

### 21. SDK 연동 테스트 안함

**실수:** SDK가 이벤트와 사용자 데이터를 올바르게 보낸다고 가정.

**결과:** 이벤트가 도착하지 않아 Campaign 트리거 안됨.

**피하는 방법:**
- 이벤트 디버거로 이벤트 확인
- 사용자 프로필에 예상 데이터 있는지 확인
- Campaign 런칭 전 연동 테스트

---

### 22. 지연된 사용자 데이터 동기화

**실수:** 사용자 생성 직후 campaign 발송.

**결과:** Notifly에 사용자가 아직 존재하지 않음. 메시지 발송 안됨.

**피하는 방법:**
- 사용자 데이터 동기화 시간 허용
- 즉시 메시지의 경우, 실시간 동기화 작동 확인
- 발송 전 사용자 존재 확인

---

### 23. 개인화 변수가 존재하지 않음

**실수:** `{{user.first_name}}` 사용하지만 사용자에게 해당 속성 없음.

**결과:** 메시지에 빈칸 또는 기본 대체값 표시.

**피하는 방법:**
- 대체값 사용: `{{user.first_name | default: "there"}}`
- 참조하는 속성이 사용자에게 있는지 확인
- 실제 사용자 데이터로 개인화 테스트

---

## 테스트 실수

### 24. 런칭 전 테스트 안함

**실수:** 테스트 발송 없이 라이브.

**결과:** 사용자가 잘못된 메시지를 받은 후 문제 발견.

**피하는 방법:**
- 항상 본인에게 먼저 테스트 발송
- Journey의 모든 경로 테스트
- 미리보기가 아닌 실제 디바이스에서 확인

---

### 25. 성공 경로만 테스트

**실수:** 모든 것이 작동하는 곳만 테스트, 엣지 케이스 안함.

**결과:** Segment에 안 맞거나, 데이터 없는 사용자 등의 문제.

**피하는 방법:**
- 메시지를 받으면 안 되는 사용자로 테스트
- 일부 데이터가 없는 사용자로 테스트
- Journey의 타임아웃과 실패 경로 테스트

## 예방 체크리스트

### Campaign/Journey 런칭 전

- [ ] 상태가 ACTIVE가 될 것인가? (테스트 모드 OFF)
- [ ] 일정이 미래이고 올바른 타임존인가?
- [ ] 이벤트 이름이 정확히 일치하나 (대소문자 구분)?
- [ ] Segment에 사용자가 있나? (미리보기 수 > 0)
- [ ] A/B 테스트 그룹 합계가 100%인가?
- [ ] 채널이 설정되고 자격증명 유효한가?
- [ ] 이벤트 분기에 타임아웃과 탈출 조건 정의?
- [ ] 실제 디바이스와 사용자로 테스트했나?
