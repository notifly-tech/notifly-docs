## 빠른 진단

깊이 들어가기 전에 먼저 확인하세요:

| 빠른 확인 | 확인 위치 |
|----------|----------|
| Campaign이 활성 상태인가? | Campaign 목록 → 상태 열 |
| 메시지가 건너뛰어졌나? | 발송 로그 → Skip 사유 |
| 사용자가 존재하나? | 사용자 조회 → ID로 검색 |
| 사용자가 해당 채널을 가지고 있나? | 사용자 프로필 → 디바이스/연락처 정보 |

---

## 의사결정 트리

이 플로우차트를 따라 문제를 진단하세요:

```
시작: 메시지가 전달되지 않음
        │
        ▼
┌─────────────────────────────┐
│ Campaign이 ACTIVE 상태인가?  │
└─────────────────────────────┘
        │
   NO ──┼── YES
   │    │
   ▼    ▼
[Campaign    ┌─────────────────────────────┐
 활성화]     │ 사용자가 segment에 있나?    │
             └─────────────────────────────┘
                     │
                NO ──┼── YES
                │    │
                ▼    ▼
         [Segment 조건   ┌─────────────────────────────┐
          확인]          │ 발송 로그 확인:              │
                         │ 메시지가 SKIPPED 되었나?     │
                         └─────────────────────────────┘
                                 │
                            YES ─┼── NO (로그에 없음)
                            │    │
                            ▼    ▼
                     [아래 Skip 사유    ┌─────────────────────┐
                      참조]             │ 채널 요구사항       │
                                        │ 확인                │
                                        └─────────────────────┘
```

## Skip 사유 설명

메시지가 건너뛰어지면 발송 로그에 skip type이 표시됩니다. 각각의 의미와 해결 방법입니다:

### `excluded_user_filter`

**의미:** 사용자가 이 채널에서 수신 거부했습니다.

**확인 방법:**
1. 제외 사용자 설정으로 이동
2. 사용자 ID 검색
3. 어떤 채널에서 제외되었는지 확인

**해결 방법:**
- 수신 거부가 의도적인 경우: 재정의 불가 (사용자 선호 존중)
- 수신 거부가 실수인 경우: 제외 목록에서 사용자 제거
- 테스트용: 다른 테스트 사용자 사용

**관련:** [제외 사용자](/ko/knowledge-base/rules/excluded-users)

---

### `forbidden_timing_filter`

**의미:** 발송 금지 시간으로 인해 메시지가 차단되었습니다.

**확인 방법:**
1. Campaign 또는 프로젝트 설정으로 이동
2. Forbidden Timing / Quiet Hours 설정 찾기
3. 시간 창과 사용자의 타임존 확인

**해결 방법:**
- 발송 금지 시간이 끝날 때까지 대기
- 발송 금지 시간 창 조정
- 발송 금지 시간을 피하도록 campaign 일정 변경
- 긴급 campaign의 경우 발송 금지 시간 비활성화 (적절한 경우)

**관련:** [발송 금지 시간](/ko/knowledge-base/rules/forbidden-timing)

---

### `fatigue_management`

**의미:** 이 campaign이 이미 이 사용자에게 허용된 최대 메시지를 발송했습니다.

**확인 방법:**
1. Campaign 설정 열기
2. Fatigue Management 섹션 찾기
3. 한도와 시간 창 확인
4. 이 사용자가 받은 메시지 수와 비교

**해결 방법:**
- 시간 창이 리셋될 때까지 대기
- 한도 증가 (적절한 경우)
- 중요 메시지의 경우 fatigue management 비활성화

**관련:** [피로도 관리](/ko/knowledge-base/rules/fatigue-management)

---

### `global_frequency_limit_filter`

**의미:** 사용자가 모든 campaign에서 허용된 최대 메시지를 받았습니다.

**확인 방법:**
1. 프로젝트 설정으로 이동
2. Global Frequency Limit 찾기
3. 일/주/월별 한도 확인
4. 사용자의 최근 메시지 이력 확인

**해결 방법:**
- 한도 창이 리셋될 때까지 대기
- 전역 한도 증가 (모든 campaign에 영향)
- 중요 campaign의 경우 "Bypass global frequency limit" 활성화

**관련:** [전역 발송 빈도 제한](/ko/knowledge-base/rules/global-frequency-limit)

---

### `noisy_filter`

**의미:** 메시지가 중복 또는 불필요한 것으로 식별되었습니다.

**확인 방법:**
- 같은 campaign이 여러 번 트리거되었나?
- 최근에 같은 메시지 내용이 발송되었나?

**해결 방법:**
- 이것은 보통 스팸 방지를 위한 올바른 동작
- 이벤트가 의도치 않게 여러 번 발생하는지 확인
- Campaign 트리거 조건 검토

---

### `connected_content_abort`

**의미:** 메시지 템플릿이 connected content (외부 API)를 사용했고, 해당 API가 발송 중단 지시를 반환했습니다.

**확인 방법:**
1. 템플릿의 connected content 설정 검토
2. 외부 API가 작동하는지 확인
3. 중단 조건 로직 확인

**해결 방법:**
- 외부 API 문제 수정
- Connected content의 중단 조건 업데이트
- Connected content 별도 테스트

## 채널별 문제

메시지가 건너뛰어지지 않았지만 여전히 전달되지 않았다면, 채널별 문제일 가능성이 높습니다:

| 채널 | 일반적인 문제 | 가이드 보기 |
|------|--------------|------------|
| 푸시 알림 | 잘못된 토큰, 권한 거부 | [푸시 장애](/ko/knowledge-base/troubleshooting/channel-failures/push-notification) |
| 이메일 | 잘못된 주소, 반송, 스팸 | [이메일 장애](/ko/knowledge-base/troubleshooting/channel-failures/email) |
| SMS | 잘못된 번호, 통신사 차단 | [SMS 장애](/ko/knowledge-base/troubleshooting/channel-failures/sms) |
| 카카오 | 친구 아님, 템플릿 미승인 | [카카오 장애](/ko/knowledge-base/troubleshooting/channel-failures/kakao) |
| LINE | 잘못된 사용자 ID, 연결 안됨 | [LINE 장애](/ko/knowledge-base/troubleshooting/channel-failures/line) |

---

## 사용자 데이터 문제

### 연락처 정보 누락

**증상:** 메시지 발송 안됨, skip 사유 없음

**확인:**
| 채널 | 필수 데이터 |
|------|------------|
| 푸시 | 유효한 디바이스 토큰 |
| 이메일 | 유효한 이메일 주소 |
| SMS | 유효한 전화번호 |
| 카카오 | 전화번호 (알림톡) 또는 친구 상태 (친구톡) |
| LINE | LINE 사용자 ID |

**해결:** 사용자 프로필에 해당 채널의 필수 데이터가 있는지 확인.

---

### 사용자를 찾을 수 없음

**증상:** 사용자 ID가 Notifly에 존재하지 않음

**확인:**
1. ID로 사용자 검색
2. 사용자 ID 형식이 앱에서 보내는 것과 일치하는지 확인
3. 사용자가 최근에 생성되었는지 확인

**해결:**
- SDK가 사용자를 올바르게 식별하는지 확인
- 사용자 ID가 올바른 형식으로 전송되는지 확인
- 사용자 데이터 동기화 대기 (몇 분 소요될 수 있음)

## Campaign 설정 문제

### Segment가 0명 반환

**증상:** Campaign 활성화되었지만 메시지 발송 안됨

**확인:**
1. Campaign 에디터에서 segment 수 미리보기
2. Segment 조건이 너무 제한적이지 않은지 확인
3. 조건이 올바른 필드 이름을 사용하는지 확인

**해결:** [세그먼트 검증](/ko/knowledge-base/troubleshooting/segment-verification) 참조

---

### 이벤트 트리거가 발동하지 않음

**증상:** 이벤트 트리거 campaign이 발송되지 않음

**확인:**
1. 이벤트 이름이 정확히 맞나? (대소문자 구분)
2. 앱/SDK가 이벤트를 보내고 있나?
3. 이벤트에 필요한 속성이 있나?

**해결:**
- 이벤트 이름 철자와 대소문자 확인
- 이벤트 디버거로 이벤트가 도착하는지 확인
- SDK 구현 확인

---

### 일정이 맞지 않음

**증상:** 예약 campaign이 실행되지 않음

**확인:**
1. Campaign 일정 설정
2. 타임존 설정
3. 예약 시간에 campaign이 활성 상태였나?

**해결:**
- 타임존이 올바른지 확인
- 예약 시간 전에 campaign이 활성화되었는지 확인
- 반복 일정 설정 검토

## 진단 체크리스트

메시지 발송 문제 조사 시 이 체크리스트를 사용하세요:

### 1단계: Campaign 상태
- [ ] Campaign이 Active 상태 (Draft나 Terminated가 아님)
- [ ] Campaign 유형이 의도한 트리거와 일치 (예약/이벤트/API)
- [ ] 일정 시간이 지남 (예약 campaign의 경우)

### 2단계: 사용자 타겟팅
- [ ] 사용자가 Notifly에 존재
- [ ] 사용자가 segment 조건에 일치
- [ ] 사용자가 해당 채널의 필수 연락처 정보 보유

### 3단계: 발송 규칙
- [ ] 사용자가 제외 사용자 목록에 없음
- [ ] 발송 금지 시간이 아님
- [ ] Campaign fatigue 한도 이내
- [ ] 전역 frequency 한도 이내

### 4단계: 채널 요구사항
- [ ] 채널이 프로젝트에 설정되어 있음
- [ ] 사용자가 해당 채널의 유효한 토큰/주소 보유
- [ ] 채널별 요구사항 충족 (채널 가이드 참조)

### 5단계: 발송 로그
- [ ] 이 사용자의 발송 로그 확인
- [ ] Skip 사유나 오류 기록
- [ ] 발송 타임스탬프와 상태 확인

## 에스컬레이션

위의 모든 것을 확인했지만 여전히 문제를 찾을 수 없는 경우:

1. **정보 수집:**
   - Campaign ID
   - 사용자 ID
   - 예상 발송 시간
   - Skip 사유 (있는 경우)
   - 설정 스크린샷

2. **최근 변경 사항 확인:**
   - Campaign 설정이 최근에 변경되었나?
   - 프로젝트 설정 (frequency limit)이 변경되었나?
   - 시스템 장애가 있었나?

3. **지원팀에 연락 시 포함할 내용:**
   - 수집된 모든 정보
   - 이미 시도한 단계
   - 예상 동작 vs 실제 동작
